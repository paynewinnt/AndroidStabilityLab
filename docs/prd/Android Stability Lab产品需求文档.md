# Android Stability Lab 产品需求文档

## 1. 文档目标

本文档用于定义 `Android Stability Lab` 的正式产品需求，统一产品定位、版本边界、功能范围、非功能要求和验收标准。

本文档回答三个问题：

1. 产品最终要做成什么
2. `V1 / V2 / V3` 各自要交付什么
3. 哪些能力必须做，哪些能力应后置

## 3. 产品定位

`Android Stability Lab` 是一个面向 Android 整机与系统稳定性场景的自动化测试与问题分析平台。

产品核心闭环如下：

`设备管理 -> 场景执行 -> 异常识别 -> 证据抓取 -> 问题聚合 -> 回归分析 -> 准入判断 -> 协作闭环`

同时，产品应保留当前项目已经具备的**性能数据监控能力**，并将其作为平台基础能力长期保留。

保留原则如下：

- 性能监控不是主定位，但仍然是核心底座
- 实时 CPU、内存、网络、FPS、功耗等采样能力继续保留
- 图表展示、会话保存、数据导出继续保留
- 性能监控能力需服务于稳定性任务执行、异常分析和报告生成

## 4. 产品目标

- 支撑 Android 稳定性测试的标准化执行
- 提升异常自动发现和自动取证能力
- 形成问题聚合、回归分析和版本判断能力
- 支撑版本准入、长稳运行和团队协作闭环

## 5. 非目标

以下内容不属于产品核心目标：

- 不作为通用功能测试管理平台
- 不作为设备运维或 MDM 平台
- 不作为线上真实用户 APM 平台
- 不采用“纯服务端执行”架构承载高频设备控制

## 6. 当前阶段与共享口径

当前项目应按“本地优先、本地部署、自看自用、产物可共享”的稳定性工作台验收，而不是按完整在线 SaaS 或多租户平台验收。

阶段判断如下：

- `V1` 执行主链已基本闭环，设备、任务、场景执行、异常检测、证据抓取和报告输出已经形成可信基础。
- `V2` 分析主链已基本闭环，问题聚合、对比、回归判断、规则治理、性能趋势和最小初步归因已经进入日常使用口径。
- `V3` 最小流程闭环已落地，当前已具备无人值守、准入对象、最小协作、本地 Web 门户和外部链路投递骨架，但仍不是完整企业级在线平台。
- 后续重点应转向生产化治理、真实平台适配、长稳产品化、规则与证据精度提升、长时运行可靠性，不再沿用早期阶段判断。

共享与部署口径如下：

- 默认以本地 CLI、本地 Web 门户、本地数据目录和本地运行产物为主。
- 报告、JSON、HTML、Markdown、分析快照、规则评审报告、准入对比和附件路径清单是当前主要共享方式。
- Web/API 可以作为本地或团队内网共享入口，但不等于已经完成多租户隔离、完整 IAM、完整在线规则发布平台、完整审批流或跨组织在线报告平台。

## 7. 用户角色

- 稳定性测试工程师：创建设备任务、执行场景、查看报告和问题
- 自动化测试工程师：维护模板、脚本和执行配置
- 开发工程师 / 系统分析工程师：查看异常详情、证据和归因线索
- 版本质量负责人：查看版本质量、回归判断和 Top Issue
- 版本经理 / 准入评审者：查看准入结论并完成评审
- 本地维护者 / 平台管理员：管理本地规则、集成配置、运行目录、部署边界和平台治理项

## 8. 版本策略

| 版本 | 定位 | 核心目标 | 核心关键词 |
| --- | --- | --- | --- |
| V1 | 稳定性任务工具 | 先打通执行闭环 | 执行、检测、取证、报告 |
| V2 | 分析与回归平台 | 先打通分析闭环 | 聚合、对比、归因、回归 |
| V3 | 量产与协作平台 | 先打通流程闭环 | 准入、协作、集成、无人值守 |

版本演进仍遵循分层建设原则：

1. `V1` 提供可信执行和证据基础
2. `V2` 提供可信分析、规则治理和回归判断
3. `V3` 在 `V1 / V2` 之上提供最小流程闭环和生产化入口

当前项目阶段判断：

- `V1 / V2` 主链已基本闭环，后续以精度、治理、保留策略、真实样本标定和少量缺口补齐为主。
- `V3` 最小流程闭环已落地，当前重点是把准入、协作、外部投递、无人值守和长稳运行从最小可用推进到生产化可靠。
- 不应再沿用早期阶段口径；更准确的表述是：`V3` 已有最小闭环，后续重点在生产化治理、真实平台适配和长稳产品化。

## 9. 通用约束

### 9.1 架构约束

长期架构为：

**本地执行代理 + 本地优先分析与共享入口 + 可选外部系统适配**

- 本地执行代理负责：设备连接、ADB 控制、场景执行、高频采样、异常初检、日志抓取、附件本地归档。
- 本地优先分析与共享入口负责：问题聚合、回归分析、看板、报告管理、准入规则、最小协作、规则评审和产物导出。
- 外部系统适配负责：在本地 outbox、webhook、API 和导出产物基础上对接 CI、IM、缺陷系统、提测平台或团队内网入口。
- 当前不按“中央平台承担所有执行和采样”验收，也不按“完整多租户在线平台”验收。

### 9.2 状态约束

通用状态模型如下：

- 设备状态：`online / offline / busy / error / quarantined`
- 任务状态：`draft / pending / running / completed / failed / cancelled`
- 执行实例状态：`pending / preparing / running / collecting / completed / failed / cancelled`
- 异常状态：`new / grouped / assigned / processing / resolved / ignored`
- 准入状态：`pending / pass / conditional_pass / fail / manually_overridden`

### 9.3 数据与治理约束

- 原始采样数据、事件数据、聚合数据、报告快照、附件数据需分层存储
- 异常指纹、回归阈值、准入规则必须可治理、可版本化、可审计
- 自动结论必须可追溯到规则版本和证据来源

### 10.1 版本摘要

`V1` 聚焦打通执行闭环，将产品从“本地性能观察工具”升级为“稳定性任务执行工具”。

- 核心闭环：`设备管理 -> 任务创建 -> 场景执行 -> 异常识别 -> 证据抓取 -> 报告输出`
- 核心交付：设备池、基础任务系统、`Monkey / 冷启动循环`、基础异常检测、基础证据抓取、统一任务报告、性能监控能力保留并接入任务链路
- 明确不做：Web 平台、中央分析后端、Top Issue、回归分析、质量门禁、协作流转、外部集成、AI 归因

## 13. 结论

`Android Stability Lab` 的阶段口径应统一为：

- `V1 / V2` 主链基本闭环，是当前可信执行、分析和准入判断的基础。
- `V3` 最小流程闭环已落地，是当前继续生产化治理和真实平台适配的起点。
- 后续重点是长稳产品化、真实外部平台适配、规则/证据治理、长时可靠性和本地部署共享体验。
- 当前共享以本地 Web、导出报告、JSON、HTML/Markdown、规则评审报告、分析快照和附件产物为主，不写成完整在线平台已完成。
