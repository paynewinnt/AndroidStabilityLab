# Android Stability Lab V1需求文档

## 1. 文档说明

本文档基于主文档 [Android Stability Lab产品需求文档](/Users/ryan.rr.penn/Develop/projects/python/AndroidStabilityLab/docs/prd/Android%20Stability%20Lab%E4%BA%A7%E5%93%81%E9%9C%80%E6%B1%82%E6%96%87%E6%A1%A3.md) 第 9 章 `V1 需求` 拆分形成，用于指导 `V1` 版本的产品设计、研发实现、测试验收与项目排期。

本文档只细化 `V1` 范围，不扩展到 `V2 / V3` 能力边界。

## 2. 版本定位与目标

### 2.1 版本定位

`V1` 的产品定位是将 Android Stability Lab 从“本地性能观察工具”升级为“稳定性任务执行工具”。

本版本必须打通如下最小闭环：

`设备管理 -> 任务创建 -> 场景执行 -> 异常识别 -> 证据抓取 -> 报告输出`

### 2.2 版本目标

`V1` 需要达成以下目标：

- 支持面向真实设备的稳定性任务创建与执行
- 支持在多设备上并行跑相同任务，并独立产出结果
- 支持对基础稳定性异常进行自动检测
- 支持在异常发生时自动保留关键证据
- 支持任务结束后自动输出统一报告
- 在任务化执行场景下复用已有性能监控能力

### 2.3 成功标准

当用户需要在提测前对某个 Android 包做一次快速稳定性筛查时，应能在一个工具中完成：

- 选择目标设备
- 选择应用与执行模板
- 启动任务并观察执行状态
- 在任务结束后查看异常和证据
- 导出报告给研发、测试或质量负责人使用

## 3. 典型用户与使用场景

### 3.1 目标用户

- 测试工程师：发起稳定性筛查任务、查看异常、导出报告
- 客户端研发：定位 crash、ANR、重启等问题，查看证据
- 质量负责人：在版本提测前查看风险摘要

### 3.2 核心场景

- 单次稳定性任务执行
- 多设备并行执行
- 版本提测前快速筛查

### 3.3 典型流程

#### 流程 A：单设备快速筛查

1. 用户连接设备并确认设备在线。
2. 用户选择目标应用、模板和执行参数。
3. 系统创建任务，并生成 1 条执行实例。
4. 模板开始执行，系统同步采集性能数据与运行日志。
5. 若识别到异常，系统自动抓取证据并关联到异常事件。
6. 任务结束后系统自动生成报告，用户查看或导出。

#### 流程 B：多设备并行验证

1. 用户选择 3 台及以上设备。
2. 系统以“一个设备一个执行实例”的方式拆分任务。
3. 各执行实例独立调度、独立记录状态、独立识别异常。
4. 用户在任务详情中查看总览，也可下钻到单设备实例查看明细。
5. 报告对整体任务进行汇总，并保留各实例的独立结果。

## 4. 范围定义

### 4.1 本版本必须交付

- 设备池管理
- 基础任务系统
- 场景模板执行
- 异常检测 `V1`
- 证据抓取 `V1`
- 任务报告 `V1`
- 历史记录与结果查询
- 已有性能监控能力保留并接入任务流

### 4.2 本版本不做

以下能力不属于 `V1` 范围：

- Web 平台
- 中央分析后端
- Top Issue 聚合
- 回归分析
- 质量门禁
- 团队协作工作流
- 外部系统集成
- AI 归因

## 5. 功能需求细化

### 5.1 设备池管理

#### 5.1.1 目标

为任务系统提供可用设备视图、设备筛选能力和设备占用控制，确保任务可稳定落到具体设备上执行。

#### 5.1.2 功能要求

系统应支持：

- 自动发现 `adb devices` 可见设备
- 展示设备在线状态与占用状态
- 保存设备基础信息
- 支持设备分组
- 支持设备标签
- 支持按条件筛选设备

#### 5.1.3 设备字段建议

设备记录建议至少包含以下字段：

| 字段 | 说明 |
| --- | --- |
| `device_id` | 系统内唯一标识 |
| `serial_no` | adb 序列号 |
| `brand` | 品牌 |
| `model` | 机型 |
| `android_version` | Android 版本 |
| `rom_version` | ROM 版本或系统构建信息 |
| `abi` | CPU 架构，建议保留 |
| `battery_level` | 当前电量 |
| `temperature` | 当前温度，若可获取 |
| `online_status` | 在线状态 |
| `occupy_status` | 空闲/占用/异常 |
| `group_name` | 所属分组 |
| `tags` | 标签集合 |
| `last_seen_at` | 最近在线时间 |

#### 5.1.4 状态建议

设备在线状态建议使用：

- `online`
- `offline`
- `unauthorized`
- `unknown`

设备占用状态建议使用：

- `idle`
- `reserved`
- `running`
- `error`

#### 5.1.5 约束

- 任务启动前必须校验所选设备处于可执行状态。
- 同一时刻一个设备只能被一个执行实例占用。
- 设备离线不应导致其他设备实例终止。

### 5.2 基础任务系统

#### 5.2.1 目标

提供面向稳定性测试的任务创建、调度、执行和生命周期管理能力，是 `V1` 的主业务骨架。

#### 5.2.2 任务对象

系统至少包含两个层级对象：

- `任务 Task`：一次用户发起的业务操作，承载公共配置与汇总结果
- `执行实例 Execution Instance`：任务在单设备上的实际执行单元

#### 5.2.3 功能要求

系统应支持：

- 创建任务
- 选择设备
- 选择目标应用
- 选择场景模板
- 配置执行时长和采样间隔
- 启动、停止、取消任务

系统应满足：

- 多设备任务需拆分为多条独立执行实例
- 每个执行实例需独立记录状态、开始时间、结束时间和执行结果

#### 5.2.4 任务字段建议

任务记录建议至少包含：

| 字段 | 说明 |
| --- | --- |
| `task_id` | 任务唯一标识 |
| `task_name` | 任务名称，默认可自动生成 |
| `package_name` | 目标应用包名 |
| `app_label` | 应用名称，建议保留展示字段 |
| `template_type` | 模板类型 |
| `template_params` | 模板参数 |
| `duration_sec` | 执行总时长 |
| `sampling_interval_sec` | 采样间隔 |
| `created_by` | 创建人，单机版可先为空或本地用户 |
| `created_at` | 创建时间 |
| `started_at` | 开始时间 |
| `ended_at` | 结束时间 |
| `status` | 任务状态 |
| `summary` | 汇总摘要 |

#### 5.2.5 执行实例字段建议

执行实例建议至少包含：

| 字段 | 说明 |
| --- | --- |
| `instance_id` | 执行实例唯一标识 |
| `task_id` | 所属任务 |
| `device_id` | 所属设备 |
| `device_snapshot` | 执行时设备快照信息 |
| `status` | 实例状态 |
| `start_at` | 实际开始时间 |
| `end_at` | 实际结束时间 |
| `result` | 执行结果 |
| `exit_reason` | 正常结束、人工停止、异常中断等 |
| `exception_count` | 异常数 |
| `first_exception_at` | 首次异常时间 |
| `report_path` | 单实例报告入口，建议保留 |

#### 5.2.6 状态流转建议

任务状态建议使用：

- `draft`
- `queued`
- `running`
- `partial_failed`
- `success`
- `failed`
- `cancelled`

执行实例状态建议使用：

- `pending`
- `precheck_failed`
- `running`
- `stopping`
- `success`
- `failed`
- `cancelled`

任务状态应由实例状态聚合得到，至少应满足：

- 全部成功时，任务标记为 `success`
- 存在部分失败但非全部失败时，任务标记为 `partial_failed`
- 全部失败或创建后无法有效执行时，任务标记为 `failed`

#### 5.2.7 执行前校验

任务开始前建议至少校验：

- 设备在线且可占用
- 目标包名合法
- 模板参数完整
- 采样间隔小于执行总时长
- 所需目录可写

若前置校验失败，应阻断对应实例启动并记录失败原因。

### 5.3 场景模板执行

#### 5.3.1 范围

`V1 MVP` 只要求上线以下模板：

- `Monkey`
- `冷启动循环`

模板应支持默认参数和有限参数改写。

#### 5.3.2 模板共性要求

- 每个模板需有默认参数集
- 每个模板需有最小必填参数集
- 模板执行前需完成设备与应用预检查
- 模板执行过程中需可被人工停止
- 模板结束后需返回明确的执行结果

#### 5.3.3 Monkey 模板建议

建议支持以下参数：

| 参数 | 说明 |
| --- | --- |
| `event_count` | 事件数或事件目标规模 |
| `throttle_ms` | 事件间隔 |
| `seed` | 随机种子，建议保留以便复现 |
| `ignore_crash` | 是否忽略 crash 后继续 |
| `ignore_timeout` | 是否忽略 timeout |
| `target_package` | 目标包 |

建议输出：

- 实际执行时长
- 实际事件数
- 是否中途退出
- 中断原因

#### 5.3.4 冷启动循环模板建议

建议支持以下参数：

| 参数 | 说明 |
| --- | --- |
| `loop_count` | 循环次数 |
| `launch_wait_ms` | 启动后观察时长 |
| `kill_before_launch` | 启动前是否强杀应用 |
| `target_activity` | 启动 Activity，可选 |
| `interval_ms` | 两次启动间隔 |

建议输出：

- 实际执行轮次
- 每轮启动结果
- 中断轮次与原因

#### 5.3.5 建议/可选实现

以下能力可作为 `V1` 的建议增强，但不应阻塞主线交付：

- 模板参数预设保存
- 最近一次任务参数复用
- 模板执行前自动清理应用数据

### 5.4 异常检测 V1

#### 5.4.1 支持识别的异常类型

系统应支持自动识别：

- `ANR`
- `Java crash`
- `native crash`
- `reboot`
- `process exit`

#### 5.4.2 异常事件字段建议

异常事件至少应包含：

| 字段 | 说明 |
| --- | --- |
| `exception_id` | 异常唯一标识 |
| `exception_type` | 异常类型 |
| `occurred_at` | 发生时间 |
| `device_id` | 设备标识 |
| `task_id` | 所属任务 |
| `instance_id` | 所属执行实例 |
| `summary` | 异常摘要 |
| `severity` | 严重级别 |

建议补充以下字段：

| 字段 | 说明 |
| --- | --- |
| `process_name` | 进程名 |
| `package_name` | 包名 |
| `pid` | 进程号，若可获取 |
| `detected_by` | 检测来源，如 logcat、系统信号 |
| `raw_marker` | 原始关键片段摘要 |
| `is_deduplicated` | 是否被归并 |

#### 5.4.3 严重级别建议

建议严重级别使用：

- `critical`：`reboot`
- `high`：`native crash`、`Java crash`、`ANR`
- `medium`：`process exit`

#### 5.4.4 检测原则

- 异常检测应以自动识别为主，不依赖人工标记。
- 检测逻辑应尽量基于已有日志、系统输出与设备状态变化。
- 单个设备的异常识别失败不得影响其他设备实例继续运行。

#### 5.4.5 去重建议

为避免报告被重复异常刷屏，建议在 `V1` 中采用轻量去重策略：

- 同一实例内
- 同一异常类型
- 同一进程或包名
- 在短时间窗口内重复出现

满足以上条件时可合并显示，但原始检测记录仍建议保留。

### 5.5 证据抓取 V1

#### 5.5.1 抓取时机

系统在异常发生时应自动抓取：

- `logcat`
- `bugreport`
- `traces.txt`
- `tombstone`

#### 5.5.2 证据对象建议

证据记录建议至少包含：

| 字段 | 说明 |
| --- | --- |
| `artifact_id` | 证据唯一标识 |
| `task_id` | 所属任务 |
| `instance_id` | 所属执行实例 |
| `exception_id` | 关联异常，可为空但建议关联 |
| `artifact_type` | 证据类型 |
| `capture_start_at` | 开始抓取时间 |
| `capture_end_at` | 结束抓取时间 |
| `file_path` | 本地路径 |
| `file_size` | 文件大小 |
| `capture_status` | 抓取状态 |
| `capture_message` | 失败原因或补充信息 |

#### 5.5.3 异常与证据关系

`V1` 建议采用以下关系模型：

- 一个异常事件可关联多个证据文件
- 一个证据文件默认归属一个执行实例
- 若证据抓取发生在实例级而非某个具体异常瞬间，可允许 `exception_id` 为空

关系使用上应满足：

- 报告中能从异常跳转到证据
- 查询中能从证据反查所属任务、实例和异常

#### 5.5.4 归档建议

证据需标准化归档，并与异常事件建立关联。

目录建议按如下维度组织：

`任务 -> 执行实例 -> 异常时间点/异常编号 -> 证据类型`

示例：

```text
artifacts/
  {task_id}/
    {instance_id}/
      {exception_id}/
        logcat.txt
        bugreport.zip
        traces.txt
        tombstone_01.txt
```

#### 5.5.5 异常场景处理

- 若某类证据不存在，应记录“未获取到”，而不是静默跳过。
- 若抓取失败，应保留失败原因并体现在报告中。
- 证据抓取失败不应导致任务主流程崩溃。

### 5.6 任务报告 V1

#### 5.6.1 目标

任务结束后应自动生成统一任务报告，服务于问题查看、问题转发和版本筛查结论输出。

#### 5.6.2 输出格式

输出格式至少支持：

- `HTML`
- `Markdown`

#### 5.6.3 报告最小结构

报告至少包含：

- 任务信息
- 设备信息
- 执行时长
- 异常数量
- 首次异常时间
- 异常明细
- 证据路径

建议按以下结构组织：

1. 报告头部
2. 任务摘要
3. 执行实例总览
4. 异常统计
5. 异常明细
6. 证据索引
7. 性能采样摘要
8. 执行备注与失败项

#### 5.6.4 报告内容建议

任务摘要建议包含：

- 任务名称、任务 ID
- 模板类型
- 目标应用
- 开始时间、结束时间、总时长
- 总设备数、成功实例数、失败实例数
- 异常总数、首个异常时间

执行实例总览建议包含：

- 设备名称/序列号
- 执行状态
- 实际运行时长
- 异常数
- 首次异常时间
- 报告或证据入口

异常明细建议包含：

- 异常类型
- 时间
- 设备
- 实例
- 摘要
- 严重级别
- 已关联证据

性能采样摘要建议至少包含：

- CPU
- 内存
- 网络
- FPS
- 功耗

`V1` 可先给出基础统计值或简要图表，不要求高级趋势分析。

### 5.7 历史记录与结果查询

#### 5.7.1 功能要求

系统应支持查询：

- 历史任务
- 执行实例
- 报告入口
- 证据路径

#### 5.7.2 查询维度建议

建议支持以下筛选维度：

- 时间范围
- 任务状态
- 模板类型
- 包名
- 设备
- 是否存在异常

#### 5.7.3 展示要求

- 历史列表应至少展示任务级摘要。
- 详情页或详情区域应可下钻到执行实例级别。
- 报告与证据应保留明确入口，不要求 `V1` 提供复杂检索分析。

### 5.8 性能监控能力保留

#### 5.8.1 保留范围

`V1` 必须保留当前已有的性能监控基础能力。

系统应继续支持：

- CPU 监控
- 内存监控
- 网络监控
- FPS 监控
- 功耗监控
- 实时图表展示
- 监控会话保存
- 监控数据导出

#### 5.8.2 融合要求

本版本要求如下：

- 性能监控能力应能被任务执行流程复用
- 监控采样结果应继续作为报告和问题分析输入
- 不再单独把“性能监控看板”作为唯一产品形态，但要保留其独立使用价值

#### 5.8.3 实现建议

- 监控模块与任务模块建议通过统一采样接口解耦。
- 单独监控模式与任务模式尽量复用同一套采样实现。
- 任务报告中优先展示摘要，不要求在 `V1` 内完成复杂跨任务趋势分析。

## 6. 非功能要求

主文档约束在 `V1` 中需具体落实为以下要求：

- 单设备异常或抓取失败不得导致主程序整体崩溃
- 至少支持 `3` 台设备并行执行稳定性任务
- 设计目标可扩展到 `8` 台设备并行
- 采样、任务调度和模板执行不应明显阻塞 CLI/Web 入口

建议进一步落实为：

- 任务调度、日志采集、证据抓取采用后台执行，不阻塞主界面交互
- 任务取消、停止操作在可接受时间内生效
- 核心状态变更需可追踪，避免出现界面状态与实际执行状态不一致
- 本地文件输出需具备基础容错，避免因单文件写入失败导致整次任务不可查看

## 7. 验收标准

### 7.1 版本级验收

以下标准全部满足，方可认为 `V1` 主线闭环成立：

- 能创建多设备稳定性任务并独立执行
- 能执行 `Monkey` 与 `冷启动循环`
- 能自动识别基础异常
- 能自动抓取基础证据
- 能自动生成统一任务报告

### 7.2 场景化验收建议

建议至少覆盖以下验收场景：

| 场景 | 预期结果 |
| --- | --- |
| 单设备执行 Monkey | 任务成功创建并完成，报告可查看 |
| 多设备并行执行 | 每个设备生成独立实例，状态互不干扰 |
| 执行中出现 Java crash | 系统识别异常并抓取至少一种证据 |
| 执行中设备离线 | 对应实例失败或中断，其他实例继续 |
| 报告导出 | 可生成 `HTML` 与 `Markdown` 报告 |
| 任务查询回看 | 可查看历史任务、实例、报告与证据路径 |

### 7.3 不通过判定示例

若出现以下情况，应视为 `V1` 不满足交付要求：

- 多设备任务未拆分为独立实例
- 异常发生后没有任何可追溯证据
- 任务结束后无法生成统一报告
- 单个设备异常导致整个应用崩溃

## 8. 实施优先级

### 8.1 P0

- 设备发现与状态展示
- 任务创建与实例拆分
- `Monkey` 与 `冷启动循环` 模板执行
- 基础异常识别
- 基础证据抓取
- 基础任务报告生成

### 8.2 P1

- 历史任务与结果查询
- 任务流程复用性能监控采样
- 异常轻量去重
- 报告中的性能摘要展示

### 8.3 建议/可选实现

以下能力可作为 `V1` 优化项，不应阻塞主线：

- 模板参数预设保存
- 失败原因的更细粒度展示
- 单实例独立报告入口

## 9. 风险与依赖

### 9.1 主要依赖

- `adb` 与设备连接稳定性
- 目标应用可被稳定拉起与控制
- 本地文件系统具备足够存储空间用于保存报告与证据
- 现有性能监控模块可复用到任务流中

### 9.2 主要风险

- 多设备并行下资源争用导致状态更新不稳定
- 不同 ROM 对日志、bugreport、tombstone 获取支持度不同
- 异常检测规则过于简单时可能出现漏报或误报
- 证据文件体积较大，影响归档速度和磁盘占用

### 9.3 风险缓解建议

- 以单设备闭环为起点，再扩展到多设备并行
- 对证据抓取失败建立明确状态和日志，而不是依赖成功路径
- 对设备离线、应用未安装、权限不足等异常前置场景增加预检查
- 报告中明确标注“未抓取到”与“抓取失败”，降低误解成本

## 10. 交付结论

`V1` 的交付核心不是做深度分析，而是先把“执行、检测、取证、报告”这条稳定性任务主链路打通并做稳。只要该闭环可在多设备场景下稳定运行，并能沉淀可追溯结果，`V1` 就达成了从观察工具向稳定性任务工具升级的目标。
