# Android Stability Lab V2需求文档

## 1. 文档目标

本文档基于主文档《`Android Stability Lab产品需求文档`》第 10 章 `V2 需求` 拆分形成，目标是将 `V2` 版本定义为一份可直接交付研发、设计、测试继续细化和实施的子需求文档。

本文档聚焦 `V2` 的版本边界、分析闭环、关键流程、功能清单、数据对象建议、规则治理、非功能要求、验收标准、实施优先级与风险控制，不扩展到 `V3` 的团队协作、完整准入流转和深度外部系统集成。

## 2. 版本定位与边界

### 2.1 版本定位

`V2` 的定位是让产品从“执行型工具”升级为“分析与回归平台”。

相较于 `V1`，`V2` 不再只回答“是否跑出了异常”，而是进一步回答：

- 问题集中在哪里
- 哪些问题最值得优先处理
- 新旧版本是否发生回归
- 异常更可能属于哪一类根因方向

### 2.2 版本目标

`V2` 应完成以下目标：

- 建立稳定的问题聚合能力，将离散异常归并为可分析的问题集合
- 建立多维度对比能力，支持版本、设备组、场景的差异分析
- 建立回归判断能力，对异常数量、出现率和关键性能指标变化给出规则化结论
- 建立分析快照能力，使分析结论可固化、可分享、可追溯
- 在问题详情中补齐证据和趋势上下文，提升研发定位效率

### 2.3 与其他版本的边界

`V2` 延续 `V1` 已具备的执行、检测、取证、报告能力，并在此基础上补充分析能力。

`V2` 明确不承担以下内容：

- 版本准入流程编排与评审闭环
- 大规模权限体系和复杂组织管理
- 全量外部平台深度打通
- AI 级自动根因分析与自动修复建议

说明：

- 若需要为后续 `V3` 预留接口或字段，可在实现中保留扩展点，但 `V2` 不以其上线为前置条件。

## 3. 核心场景

### 3.1 版本回归判断

测试或质量负责人选择两个版本的执行结果集，按相同场景和设备范围进行比较，输出：

- 新增问题
- 加重问题
- 已消失问题
- 关键性能指标恶化项
- 当前回归结论

### 3.2 问题聚合分析

研发或分析工程师查看一段时间内的异常记录，系统将相似异常自动聚合为问题项，并按频次、影响范围、严重级别进行排序，输出 `Top Issue` 列表。

### 3.3 异常定位辅助

研发查看问题详情时，能同时看到：

- 异常摘要
- 指纹命中依据
- 同类问题历史分布
- 异常前后关键性能趋势
- 已抓取证据入口
- 初步归因方向

### 3.4 业务遍历场景接入

自动化测试工程师可将 `uiautomator2`、`adb` 输入脚本、自定义业务遍历脚本接入平台，使业务路径执行结果可以进入统一的聚合与回归分析链路。

## 4. 分析闭环

### 4.1 闭环定义

`V2` 的最小分析闭环为：

`执行结果接入 -> 异常标准化 -> 指纹生成 -> 问题聚合 -> 多维对比 -> 回归判断 -> 快照固化 -> 问题详情追溯`

### 4.2 闭环说明

- 执行结果接入：复用 `V1` 任务、实例、异常和证据输出，接入业务遍历等新增执行来源
- 异常标准化：将不同模板、不同设备、不同脚本上报的异常记录统一为标准事件结构
- 指纹生成：基于异常类型、关键信息、堆栈特征、进程信息、场景上下文生成稳定指纹
- 问题聚合：将同类异常归并为问题，并形成跨执行实例、跨任务、跨版本的聚合视图
- 多维对比：支持版本、设备组、场景等视角下的问题和指标对比
- 回归判断：通过规则和阈值对差异结果打标，产出新增、恶化、持平、改善等状态
- 快照固化：将重要分析结果固化为可追溯快照，避免因后续规则变化导致结论漂移
- 问题详情追溯：从聚合问题可反查到原始事件、证据文件、趋势图和规则版本

## 5. 关键流程

### 5.1 聚合流程

建议按以下流程实现问题聚合：

1. 接收来自任务执行侧的异常事件和性能异常事件。
2. 对异常进行标准化补全，统一时间、设备、版本、场景、应用、异常类型等核心字段。
3. 按异常类型进入对应的指纹策略。
4. 生成主指纹和候选相似指纹。
5. 根据指纹命中、时间窗口、设备与场景上下文进行归并。
6. 生成或更新聚合问题对象，回填最新出现时间、影响范围、计数与样本列表。
7. 对聚合结果执行排序计算，生成 `Top Issue` 视图数据。

建议：

- 不同异常类型使用不同权重的指纹策略，避免“一刀切”导致误归并。
- 首版可优先保障“宁可少合并，也不要错误合并”，人工修正能力应早于复杂自动算法。

### 5.2 回归分析流程

建议按以下流程实现回归分析：

1. 选择比较对象：版本 A / B，或设备组 A / B，或场景 A / B。
2. 根据对比条件筛选可比数据集，校验样本量、执行时长、覆盖场景是否满足基本可比性。
3. 对聚合问题进行维度对齐，生成同类问题映射关系。
4. 计算异常数量、触发率、设备覆盖率、场景覆盖率、首次出现时间等差异。
5. 计算 CPU、内存、FPS、功耗等关键指标的统计差异。
6. 按阈值规则输出每个问题、每项指标、整体结果的回归判断。
7. 产出可保存的回归结论快照，并记录当时使用的规则版本和样本范围。

### 5.3 对比分析流程

系统应支持以下对比视角：

- 版本对比：比较不同 ROM、系统版本、应用版本的异常与性能变化
- 设备组对比：比较不同 SoC、品牌、机型、内存规格下的问题分布
- 场景对比：比较 `Monkey`、冷启动循环、业务遍历等不同场景的风险差异

对比结果建议统一输出：

- 样本说明
- 问题变化摘要
- 关键指标变化摘要
- 回归结论
- 不可比说明或数据风险提示

### 5.4 初步归因流程

`V2` 的归因应限定为规则式初步归因，不承诺深度智能分析。

建议流程如下：

1. 依据异常类型选择归因规则集。
2. 综合堆栈关键词、进程名、系统组件、性能异常窗口、场景步骤和证据摘要进行判定。
3. 输出“疑似归因方向”和“命中依据”。
4. 标注可信度等级，例如：`high / medium / low`。
5. 允许人工修正归因方向并记录修正人、修正时间和原因。

归因方向建议限定在以下范围：

- 应用侧逻辑异常
- Framework / 系统服务异常
- 驱动 / 硬件相关异常
- 资源压力相关
- 图形渲染或显示链路相关
- 暂无法判断

## 6. 功能清单

### 6.1 场景编排增强

系统应支持：

- 多步骤场景配置
- 步骤顺序控制
- 步骤失败后的中断、跳过、重试策略
- 步骤级超时配置
- 批量排期
- 长稳任务队列

建议补充：

- 每个步骤可记录开始时间、结束时间、执行耗时、步骤结果和失败原因
- 步骤日志应可用于后续问题详情页定位“异常出现于哪一步”

### 6.2 业务遍历与自动化接入

系统应支持：

- `uiautomator2`
- `adb` 输入脚本
- 自定义业务遍历脚本
- 外部脚本执行回调

为了便于统一分析，接入侧至少应上报：

- 执行入口名称
- 场景标识
- 步骤标识
- 执行开始与结束时间
- 成功 / 失败状态
- 失败摘要
- 关联截图、日志或附件路径

### 6.3 异常检测增强

系统应新增识别：

- `system_server crash`
- `watchdog`
- `freeze / 黑屏`
- `OOM 风险`
- `内存持续上涨`
- `卡顿 / 帧耗时恶化`

建议细化如下：

- `freeze / 黑屏` 应区分“屏幕无刷新”“输入无响应”“界面黑屏”这三类观测信号
- `OOM 风险` 应以阈值预警为主，不与最终 `Java OOM crash` 混为一类
- `内存持续上涨` 应至少支持按滑动窗口识别趋势
- `卡顿 / 帧耗时恶化` 应支持与场景步骤时间轴对齐

### 6.4 证据抓取增强

系统应按需支持：

- `dropbox`
- `dumpsys activity`
- `dumpsys meminfo`
- `dumpsys SurfaceFlinger`
- `perfetto`

建议按“异常类型 -> 抓取策略”配置：

- 应用或系统崩溃优先抓取 `dropbox`、`bugreport`、关键日志
- 内存异常优先抓取 `dumpsys meminfo`
- 图形或卡顿异常优先抓取 `SurfaceFlinger` 和可选 `perfetto`
- 长稳任务中应支持按频率限制高成本证据抓取，避免影响执行稳定性

### 6.5 问题聚合与 Top Issue

系统应支持：

- 问题指纹
- 问题归并
- 按版本聚合
- 按设备聚合
- 按场景聚合
- `Top Issue` 排序

`Top Issue` 建议综合以下因素排序：

- 触发次数
- 受影响设备数
- 受影响场景数
- 严重级别
- 是否新增或回归
- 最近出现时间

说明：

- 排序公式可配置，但首版应提供稳定默认策略。

### 6.6 回归分析与初步归因

系统应支持：

- 版本 A / B 对比
- 设备组对比
- 场景对比
- 回归阈值配置
- 自动回归判断
- 规则式初步归因

建议首版回归结果至少包含：

- `new`
- `worsened`
- `unchanged`
- `improved`
- `gone`
- `insufficient_data`

### 6.7 报表与分析快照

系统应支持固化：

- 版本对比快照
- 场景对比快照
- 回归结论快照
- `Top Issue` 快照

每类快照建议至少包含：

- 快照标题
- 创建时间
- 创建人或创建来源
- 数据时间范围
- 数据筛选条件
- 规则版本
- 关键结论
- 问题列表摘要
- 快照详情入口

### 6.8 性能趋势分析

系统应支持：

- 按任务查看性能趋势
- 按版本比较 `CPU / 内存 / FPS / 功耗` 等关键指标
- 在问题详情中查看异常前后关键指标变化
- 在报告中引用关键趋势图和时间点标注

建议补充：

- 趋势图支持统一时间轴，能够对齐异常点、步骤点、证据抓取点
- 关键指标建议同时提供平均值、峰值、P95 或最大连续恶化时长等统计方式

## 8. 快照与规则治理

### 8.1 快照治理要求

系统应满足：

- 快照创建后内容不可被静默覆盖
- 快照需保留数据范围、筛选条件和规则版本
- 快照中的核心结论应可追溯到对应问题和原始事件
- 快照应支持再次查看，不依赖实时重算才能展示核心信息

### 8.2 规则治理要求

系统应满足：

- 指纹规则、聚合规则、回归阈值规则、归因规则必须版本化
- 规则变更需记录变更人、变更时间、变更说明
- 规则生效时间需明确，避免历史结果解释混乱
- 人工修正规则结果时需保留审计轨迹

### 8.3 人工修正能力

为降低自动聚合误差，建议 `V2` 至少提供以下人工修正能力：

- 合并两个问题
- 拆分误归并的问题
- 调整问题标题和严重级别
- 调整疑似归因方向
- 标记忽略或误报

说明：

- 人工修正能力属于分析治理范畴，仍在 `V2` 合理边界内；但不扩展为完整工单协作流转。

## 9. 非功能要求

### 9.1 一致性与可复现性

- 同一批输入数据在相同规则版本下应得到一致的聚合结果
- 回归分析在相同筛选条件下应得到稳定结论
- 快照查看结果应与创建时的核心结论保持一致

### 9.2 性能要求

- 常用问题列表、`Top Issue` 列表、版本对比结果应具备缓存或预计算机制
- 单次分析页面不应因实时重算导致明显不可用
- 高成本分析项如 `perfetto` 解析应支持异步处理

### 9.3 可靠性要求

- 单次规则计算失败不得影响原始任务和事件数据可用性
- 某一类证据抓取失败时，系统仍应保留异常事件并标记证据缺失
- 聚合、对比、快照生成任务应具备失败重试和状态可观测能力

### 9.4 可观测性要求

平台应具备以下自监控指标：

- 任务失败率
- `ADB` 失败率
- 证据抓取失败率
- 规则计算失败率
- 聚合命中率
- 人工修正率
- 快照生成耗时

### 9.5 可扩展性要求

- 数据模型需支持后续新增异常类型与新证据类型
- 对比分析维度应可扩展到版本、设备、场景之外的更多标签维度
- 规则引擎应支持新增规则类型而不重构全链路

## 10. 验收标准

### 10.1 功能验收

`V2` 至少应满足以下验收项：

- 能从多次执行结果中聚合同类问题
- 能输出 `Top Issue` 列表
- 能完成版本维度对比并给出回归结论
- 能完成设备维度和场景维度对比
- 能在问题详情中展示关键证据和异常前后趋势变化
- 能固化版本对比、场景对比、回归结论和 `Top Issue` 快照
- 能给出规则式初步归因及命中依据

### 10.2 数据验收

- 任一聚合问题都可追溯到原始异常事件
- 任一回归结论都可追溯到比较范围和规则版本
- 任一快照都可追溯到创建时间、筛选条件和数据范围

### 10.3 稳定性验收

- 聚合规则在重复计算中结果稳定
- 常用分析页面在正常数据量下可稳定打开
- 异常证据缺失或部分解析失败时不影响主体分析链路

## 11. 实施优先级

### 11.1 P0

- 异常标准化模型
- 问题指纹与基础聚合
- `Top Issue` 列表
- 版本 / 设备 / 场景对比基础能力
- 回归阈值配置与自动回归判断
- 分析快照固化
- 问题详情页中的证据与趋势联动

### 11.2 P1

- 多步骤场景编排增强
- `uiautomator2` 与自定义业务遍历接入
- 增强异常检测
- 增强证据抓取
- 规则式初步归因
- 人工修正聚合结果

### 11.3 P2

- 更丰富的排序策略
- 更细粒度的趋势统计方式
- 更复杂的相似问题匹配策略

建议：

- 若资源有限，应优先打通“问题聚合 + 对比 + 回归 + 快照”这一条主分析链路，再扩展检测种类和复杂编排能力。

## 12. 主要风险与应对

### 12.1 指纹不稳定导致误聚合

风险：

- 同类问题被拆散，无法形成有效聚合
- 不同问题被错误合并，误导研发判断

应对建议：

- 首版先按异常类型拆分策略
- 引入人工合并 / 拆分能力
- 用历史样本持续校验指纹命中质量

### 12.2 对比样本不可比导致误判回归

风险：

- 样本量过小或覆盖不一致，造成错误回归结论

应对建议：

- 在分析前校验样本量、时长、场景覆盖和设备覆盖
- 对不满足条件的结果明确标记 `insufficient_data`

### 12.3 高成本证据影响执行稳定性

风险：

- 过度抓取 `bugreport`、`perfetto` 等大文件导致执行抖动

应对建议：

- 基于异常类型按需抓取
- 引入频率限制和开关配置
- 将重处理任务异步化

### 12.4 规则频繁变化导致历史结果失真

风险：

- 规则更新后，新旧分析结果难以解释

应对建议：

- 所有分析结论绑定规则版本
- 快照固化结论，不依赖实时重算覆盖历史

- 为问题详情补充异常时间轴视图
