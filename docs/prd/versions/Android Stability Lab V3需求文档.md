# Android Stability Lab V3需求文档

## 1. 文档定位

本文档是 [`Android Stability Lab产品需求文档`](../Android%20Stability%20Lab%E4%BA%A7%E5%93%81%E9%9C%80%E6%B1%82%E6%96%87%E6%A1%A3.md) 中第 11 章 `V3` 需求的独立子文档，用于将主文档中的版本边界细化为可执行的产品、研发、设计、测试输入。

本文档只细化 `V3` 范围内能力，不扩展超出主文档的新版本边界。

`V3` 的定位保持不变：

- 从“分析平台”升级为“量产与协作平台”
- 以 `7x24` 稳定运行、版本准入、团队协作、外部集成为核心
- 建立从提测触发到准入结论、从异常发现到责任闭环的流程能力

## 2. 版本目标

`V3` 需要在 `V1` 执行闭环、`V2` 分析闭环基础上，补齐流程闭环，使平台具备被版本团队日常使用和依赖的能力。

本版本重点解决以下问题：

- 如何让长稳任务 `7x24` 无人值守连续运行
- 如何让稳定性结果进入版本准入流程
- 如何让问题从发现、认领、处理到结论沉淀形成闭环
- 如何与提测平台、告警渠道、缺陷系统、CI/CD 建立标准对接

本版本交付结果应体现为：

- 稳定性平台成为版本提测后的标准检查环节之一
- 业务团队可以通过 Web 控制台完成任务查看、问题协作、准入评审
- 平台具备基本可运营、可审计、可持续运行能力

## 3. 适用角色

- 稳定性测试工程师：管理长稳任务、查看异常、跟踪执行质量
- 自动化测试工程师：维护执行编排、轮转策略、集成触发配置
- 开发工程师 / 系统分析工程师：认领问题、查看证据、沉淀处理结论
- 版本质量负责人：查看版本质量趋势、审核准入风险
- 版本经理 / 准入评审者：基于平台结论进行放行或条件放行决策
- 平台管理员：配置角色、规则、订阅、外部集成、自监控治理项

## 4. 核心场景

### 4.1 提测自动触发

版本在提测平台提交后，平台自动创建对应稳定性任务，执行预定义场景，并输出可审计的准入结论。

### 4.2 7x24 长稳运行

平台持续调度长稳任务，在设备波动、任务失败、代理异常等情况下自动恢复、补位或隔离，保证长周期运行。

### 4.3 准入评审

版本质量负责人和评审者查看自动生成的准入结论、风险项、例外项和人工覆盖记录，形成正式评审结果。

### 4.4 问题协作闭环

问题从平台自动发现开始，经问题归并、认领、评论、责任确认、结论沉淀，最终进入已解决、忽略或保留观察状态。

## 5. 版本范围总览

`V3` 范围包含以下能力域：

1. `7x24` 无人值守运行
2. 质量门禁与准入中心
3. 团队协作与问题流转
4. 外部系统集成
5. Web 控制台
6. 通知与订阅
7. 性能基线与门禁辅助
8. 非功能治理与运行保障

其中，`V3` 仍遵循主文档的长期架构约束：

- 本地执行代理负责设备连接、场景执行、高频采样、证据抓取
- 中央分析平台负责聚合分析、规则评估、协作流转、看板与集成

## 6. 功能需求

### 6.1 7x24 无人值守

#### 6.1.1 目标

平台应支持长稳任务连续运行，不依赖人工频繁值守，并在常见故障场景下维持任务池稳定。

#### 6.1.2 核心能力

系统应支持：

- 长稳任务自动轮转
- 多设备自动补位
- 失败后自动重试
- 设备异常自动恢复或隔离
- 每日自动汇总

系统建议进一步细化为以下能力：

- 按场景配置轮转策略，如固定时长轮转、异常后轮转、每日定时轮转
- 设备池按优先级补位，优先选择 `online` 且未占用设备
- 对执行失败支持分级重试，如任务重试、步骤重试、设备重连后重试
- 对连续失败设备执行自动隔离，将设备状态置为 `quarantined`
- 对隔离设备保留自动恢复探测机制，恢复成功后可重新入池

#### 6.1.3 调度规则

系统至少应支持以下调度规则：

- 长稳任务可配置并发数、运行窗口、轮转周期、单实例最长时长
- 同一设备同一时刻只能承载一个执行实例
- 调度器应避免频繁在异常设备上重复分配任务
- 当设备池不足时，任务状态应进入可追踪的等待或降级状态，而不是静默失败

建议补充：

- 可选实现：支持设备池分层调度，如核心机型优先、探索机型补充
- 可选实现：支持夜间与工作时段使用不同轮转策略

#### 6.1.4 故障恢复

系统应定义至少以下恢复动作：

- `adb` 断连时自动重连
- 执行代理心跳丢失时停止继续派发并告警
- 任务实例异常退出后自动拉起补偿实例
- 设备出现高温、低电、反复离线时自动隔离并记录原因

故障恢复记录至少包含：

- 故障类型
- 发生时间
- 影响范围
- 自动恢复动作
- 恢复结果
- 是否需要人工介入

#### 6.1.5 每日汇总

系统应自动生成长稳日报，至少包含：

- 当日执行任务数
- 总运行时长
- 设备在线率
- 失败率
- 异常类型分布
- Top Issue 变化
- 中断任务明细
- 隔离设备清单

### 6.2 质量门禁与准入中心

#### 6.2.1 目标

将稳定性结果转化为明确、可审计、可复现的版本准入结论。

#### 6.2.2 规则模型

系统应支持：

- 版本准入规则
- 异常上限门槛
- 回归阈值门槛
- 场景覆盖门槛
- 自动准入结论：`pass / conditional_pass / fail`

建议将规则拆分为以下层次：

- 全局规则：适用于全部版本线的统一门槛
- 产品线规则：适用于特定项目、机型组或业务线
- 临时规则：仅在当前提测批次或灰度周期内生效

每条规则至少包含：

- 规则名称
- 规则版本
- 适用范围
- 指标定义
- 判断方式
- 阈值
- 生效时间
- 创建人和修改人

#### 6.2.3 准入评估输入

准入评估至少使用以下输入：

- 任务执行结果
- 异常聚合结果
- Top Issue 列表
- 回归分析结论
- 场景覆盖情况
- 关键性能指标基线比较结果

其中性能结果在 `V3` 中只作为风险辅助项，不替代稳定性结论。

#### 6.2.4 准入输出

准入中心至少输出：

- 自动结论
- 触发规则列表
- 不通过原因
- 风险提示项
- 覆盖不足项
- 人工覆盖记录
- 最终评审意见

#### 6.2.5 人工覆盖与审计

系统应允许有权限的角色对自动结论进行人工覆盖，但必须满足：

- 覆盖人身份可追溯
- 覆盖原因必填
- 覆盖前后结论同时保留
- 关联证据与评论可追溯
- 审计记录不可静默删除

### 6.3 团队协作与问题流转

#### 6.3.1 目标

让平台内的问题记录成为稳定性问题的统一协作入口，而不是仅输出静态报告。

#### 6.3.2 用户与角色

系统应支持基础用户与角色能力，满足以下最小需要：

- 查看权限
- 认领权限
- 评论权限
- 状态流转权限
- 准入审核权限
- 管理配置权限

本版本不追求复杂的企业级权限系统，但需覆盖版本评审和问题协作所需的最小控制面。

#### 6.3.3 问题流转模型

问题状态应兼容主文档中的通用状态模型：

- `new`
- `grouped`
- `assigned`
- `processing`
- `resolved`
- `ignored`

建议在界面层补充派生视图标签，而不新增底层冲突状态，例如：

- 待认领
- 待确认
- 待回归验证
- 已沉淀

#### 6.3.4 协作动作

系统应支持：

- 问题认领
- 状态流转
- 评论记录
- 责任归属
- 结论沉淀

进一步要求如下：

- 评论支持 `@人` 和引用附件或报告链接
- 问题支持标记责任团队、责任人、影响范围
- 解决结论需支持结构化字段，如根因分类、修复版本、规避方案
- 被忽略的问题必须填写忽略原因

#### 6.3.5 缺陷联动

平台与缺陷系统集成后，应支持：

- 从问题详情创建缺陷
- 回填外部缺陷单号
- 同步缺陷状态快照
- 在问题关闭前校验缺陷是否已到达可接受状态

建议补充：

- 可选实现：根据问题模板自动生成缺陷摘要
- 可选实现：按责任团队自动分配默认缺陷模板

#### 6.3.6 结论沉淀

系统应支持将历史处理结论沉淀为可复用知识，包括：

- 常见指纹对应根因方向
- 建议排查路径
- 是否属于已知问题
- 建议忽略条件

该能力在 `V3` 以结构化结论沉淀为主，不要求引入 AI 自动生成结论。

### 6.4 外部系统集成

#### 6.4.1 目标

通过标准化事件和接口，将平台嵌入既有研发流程。

#### 6.4.2 集成范围

系统应支持：

- 提测平台集成
- IM 告警集成
- 缺陷系统集成
- CI/CD 集成

#### 6.4.3 触发与回传链路

建议采用统一事件模型，至少覆盖：

- 提测创建事件
- 任务开始事件
- 任务失败事件
- 准入完成事件
- 问题创建事件
- 长稳中断事件

每类事件至少应支持：

- 唯一事件标识
- 事件时间
- 关联版本或任务标识
- 重试机制
- 幂等保护
- 回传结果记录

#### 6.4.4 提测平台集成

系统应支持：

- 提测单触发稳定性任务
- 读取提测版本基础信息
- 回写稳定性执行状态
- 回写准入结论或准入链接

#### 6.4.5 IM 告警集成

系统应支持向企业 IM 渠道发送标准通知，至少覆盖：

- 任务失败
- 异常高发
- 长稳中断
- 准入结论生成
- 日报 / 周报摘要

#### 6.4.6 CI/CD 集成

系统应支持：

- 构建完成后触发平台任务
- 平台将任务结果和准入结论回传流水线
- 根据门禁结论阻断或放行后续流程

本能力只要求与标准 CI/CD 流程打通，不要求平台替代构建系统。

### 6.5 Web 控制台

#### 6.5.1 目标

提供统一的日常使用入口，使测试、开发、版本质量角色无需依赖本地工具即可查看状态与完成协作。

#### 6.5.2 一级入口

系统应提供至少以下 Web 页面：

- 首页看板
- 任务大厅
- 问题中心
- 报告中心
- 准入中心
- 集成中心

#### 6.5.3 页面要求

首页看板至少展示：

- 今日执行概览
- 设备在线率
- 长稳运行状态
- Top Issue
- 待处理风险
- 最近准入结论

任务大厅至少支持：

- 任务筛选
- 任务状态查看
- 失败实例定位
- 长稳任务运行态查看
- 批量查看轮转与重试记录

问题中心至少支持：

- 问题列表筛选
- 指纹聚合视图
- 问题详情
- 责任归属信息
- 评论与流转记录
- 缺陷联动入口

报告中心至少支持：

- 历史报告检索
- 长稳日报 / 周报查看
- 版本准入报告查看
- 报告快照分享链接

准入中心至少支持：

- 准入单列表
- 规则命中详情
- 风险项摘要
- 人工覆盖操作
- 审计记录查看

集成中心至少支持：

- 集成配置列表
- 最近回调记录
- 回调失败重试记录
- 告警渠道状态

#### 6.5.4 体验要求

Web 控制台需满足：

- 核心页面默认加载最近可用数据，不依赖用户复杂查询才能看到状态
- 长列表支持筛选、排序、分页和常用条件保存
- 异常、报告、准入之间可相互跳转
- 关键结论与风险项应突出展示，避免埋在原始数据明细中

### 6.6 通知与订阅

#### 6.6.1 目标

让不同角色在合适时间接收到合适粒度的通知，减少人工巡检成本。

#### 6.6.2 通知范围

系统应支持：

- 任务失败通知
- 异常高发通知
- 准入结论通知
- 长稳中断通知
- 日报 / 周报摘要通知

#### 6.6.3 订阅维度

建议支持以下订阅维度：

- 按产品线订阅
- 按版本线订阅
- 按任务类型订阅
- 按设备组订阅
- 按异常等级订阅
- 按角色默认订阅

#### 6.6.4 通知治理

系统应具备以下治理能力：

- 相同事件合并去重
- 高频告警抑制
- 通知失败重试
- 通知日志留存
- 支持临时静默窗口

建议补充：

- 可选实现：支持日报和周报的订阅模板
- 可选实现：支持面向管理层的简版摘要模板

### 6.7 性能基线与门禁辅助

#### 6.7.1 目标

在不偏离“稳定性优先”定位前提下，将关键性能指标纳入版本评估参考。

#### 6.7.2 能力要求

系统应支持：

- 关键性能指标基线
- 版本间性能退化提示
- 在准入报告中引用关键性能指标
- 将性能异常作为风险提示项，而不是替代稳定性结论

关键指标建议至少包含：

- CPU
- 内存
- FPS
- 功耗
- 冷启动耗时

#### 6.7.3 结果使用原则

- 性能异常默认归类为“风险提示”，不直接导致稳定性准入失败，除非规则明确配置
- 准入报告中应区分“稳定性结论”和“性能风险”
- 性能趋势图应支持定位到异常前后时间段

## 7. 非功能要求

### 7.1 连续运行与稳定性

平台应支持长周期连续运行，至少满足：

- 调度器和关键后台任务可持续运行
- 单点任务失败不应导致整体调度停摆
- 执行代理异常应被快速发现并阻断继续派发
- 系统重启后能恢复关键任务上下文和最近状态

### 7.2 可观测性

平台应具备自监控能力，至少包括：

- 调度器心跳
- 执行代理心跳
- 设备在线率
- 任务失败率
- 证据抓取失败率
- 外部回调失败率
- 通知发送失败率

建议将自监控结果纳入首页看板和运维告警。

### 7.3 存储与数据治理

原始采样、聚合数据和附件应分层存储，并满足：

- 任务明细与报告快照可追溯
- 附件引用关系清晰
- 热数据与冷数据支持分层保留
- 可配置数据保留周期

### 7.4 审计与合规

质量门禁规则和人工覆盖结论应可审计，至少记录：

- 谁创建或修改了规则
- 谁执行了人工覆盖
- 覆盖原因是什么
- 何时生效
- 影响了哪些版本或准入单

### 7.5 Web 与接口性能

建议以下性能目标作为研发验收基线：

- 常用列表页在常规筛选条件下可在 `3` 秒内返回首屏结果
- 准入详情、问题详情、报告详情需支持按需加载大字段和附件索引
- 外部回调接口需支持失败重试且避免重复写入

### 7.6 安全与权限

本版本需满足最小安全要求：

- Web 登录态和关键操作需具备权限校验
- 集成密钥、Webhook 凭证不得明文暴露在前端页面
- 审计日志与操作日志应区分普通用户和管理员可见范围

## 8. 验收标准

### 8.1 7x24 运行验收

- 长稳任务可无人值守连续运行，并支持自动轮转
- 设备异常、离线、代理异常时可自动恢复或隔离
- 每日能自动生成长稳汇总

### 8.2 准入验收

- 提测版本可自动触发稳定性任务
- 平台可根据规则输出自动准入结论
- 人工覆盖结论后可保留审计记录

### 8.3 协作闭环验收

- 团队可在平台内完成问题分派、评论、状态流转和结论沉淀
- 问题可与缺陷系统建立关联
- 责任归属和处理历史可追溯

### 8.4 Web 控制台验收

- 首页看板、任务大厅、问题中心、报告中心、准入中心、集成中心可支撑日常使用闭环
- 关键页面可完成查询、跳转、筛选和核心操作

### 8.5 通知与集成验收

- 任务失败、异常高发、长稳中断、准入结论可触发通知
- 提测平台、IM、缺陷系统、CI/CD 中至少完成主流程联通

## 9. 实施优先级

### 9.1 P0

- `7x24` 无人值守运行最小闭环
- 质量门禁规则与准入结论输出
- 首页看板、任务大厅、问题中心、准入中心基础版
- 任务失败、长稳中断、准入结论通知
- 提测平台与 IM 主链路打通

### 9.2 P1

- 问题认领、评论、责任归属、结构化结论沉淀
- 缺陷系统联动
- 日报 / 周报摘要
- 集成中心与回调记录
- 性能基线风险提示纳入准入报告

### 9.3 P2

- 更细的设备池调度策略
- 订阅模板与静默治理增强
- 更丰富的知识沉淀模板
- 建议/可选实现项落地

## 10. 依赖与风险

### 10.1 关键依赖

- `V1` 的执行稳定性和证据抓取能力可稳定复用
- `V2` 的问题聚合、Top Issue、回归分析已具备可复用结果
- 执行代理具备心跳、重连和失败上报机制
- 外部平台可提供可用的回调或接口能力

### 10.2 主要风险

- 设备质量波动会直接影响 `7x24` 运行效果
- 若 `V2` 聚合质量不足，`V3` 准入结论可信度会下降
- 多系统集成若无统一事件模型，后续维护成本会快速上升
- 若权限与审计能力缺失，准入结果难以在正式评审中被采信

### 10.3 风险缓解建议

- 优先补强设备隔离、恢复、自监控能力
- 准入规则采用版本化管理，避免口径漂移
- 先打通主链路集成，再扩展次要系统
- Web 控制台优先交付核心决策页面，避免前期页面铺得过宽

## 11. 建议的研发拆解

为便于研发并行推进，建议按以下工作包拆解：

1. 调度与运行保障：长稳轮转、补位、重试、隔离、自监控
2. 准入与规则引擎：规则配置、结论计算、人工覆盖、审计
3. 协作闭环：问题流转、评论、责任归属、结论沉淀
4. Web 控制台：看板、任务、问题、准入、报告、集成
5. 集成与通知：提测平台、IM、缺陷系统、CI/CD、订阅治理

该拆解为实施建议，不改变主文档定义的版本边界。
