# 性能采样子系统方案

## 1. 文档目标

本文档定义 `Android Stability Lab` 的性能采样子系统方案，说明采样目标、后端类型、数据流、接入边界和验收标准。

本文档只讨论性能采样子系统，不讨论整个产品重构。

## 2. 方案结论

性能采样能力应作为稳定性执行链路的一等基础能力存在，通过统一 `monitoring adapter` 接入任务、Run、报告、性能采样页和准入风险判断。

当前推荐形态：

- `adb_collector`：默认基础链路，适合通用 CPU、内存、网络等快照采样。
- `solox`：面向无 Root 设备的一站式性能采样，适合补充 network、battery、fps、gpu 等指标。
- `perfetto`：面向深度定位的 trace sidecar，适合辅助系统级 trace、network packets 和性能问题下钻。

## 3. 子系统目标

性能采样子系统需要满足：

- 能随 Run 启停，并把采样结果挂到具体执行实例。
- 能输出统一 snapshot contract，供 Web、CLI、报告和准入服务复用。
- 能记录 `monitoring_backend / key_metrics / trace_artifact_path / snapshot_path`。
- 能在采样失败时降级，不阻断稳定性主任务的结果沉淀。
- 能区分基础快照、扩展性能指标和 trace 产物。

## 4. 分层设计

### 4.1 后端选择层

负责解析配置和单次执行覆盖参数：

- 全局配置：`config/monitoring.json`
- 单次覆盖：`--monitoring-backend adb_collector|solox|perfetto|auto|disabled`
- Web 执行入口：任务大厅的执行 Run 表单

### 4.2 采样会话层

负责：

- 创建采样 session。
- 绑定 `task_id / run_id / instance_id / device_id / package_name`。
- 保存 session 状态、采样目录和 trace 目录。
- 在执行结束时停止采样并写入最终 metadata。

### 4.3 指标采集层

按后端采集不同指标：

- 基础指标：CPU、Memory、Network。
- SoloX 扩展指标：Battery、FPS、GPU、Network。
- Perfetto trace：trace 文件、trace 状态、trace 大小、采集模式。

### 4.4 结果持久化层

负责把采样结果写入：

- 执行实例 metadata。
- monitoring snapshot JSON。
- runtime 产物目录。
- 报告、Web `/performance` 和 `/runs/<run_id>` 下钻入口。

## 5. 数据输出要求

每次采样至少应尽量输出：

- `backend`
- `device_id`
- `package_name`
- `captured_at`
- `key_metrics`
- `snapshot_path`
- `trace_artifact_path`
- `artifacts`
- `warnings`

如果后端不支持某个指标，应明确输出缺失原因，而不是显示含义不清的空值。

## 6. Web 展示要求

Web 层应提供：

- 首页最近性能采样摘要卡。
- `/performance` 聚合最近监控快照。
- `/tasks` 和 Run 详情展示 `monitoring_backend / key_metrics / trace link`。
- 对 `adb_collector / solox / perfetto` 的差异做用户可理解的说明。

## 7. 验收标准

性能采样子系统至少应满足：

- `adb_collector` 能随 Run 正常采样并落盘 snapshot。
- `solox` 能在依赖可用时输出扩展指标，依赖不可用时给出明确错误。
- `perfetto` 能输出 trace artifact，必要时支持 stdout fallback。
- Web `/performance` 能看到最近采样、后端类型、关键指标和 trace 链接。
- 采样失败不应导致稳定性任务结果丢失。

## 8. 后续增强

后续可继续补：

- Perfetto trace 自动解析。
- SoloX 指标质量校验。
- 性能风险阈值治理页。
- 按设备、包名、场景维度的阈值标定。
- 与 `AdmissionCase` 的性能风险证据链进一步联动。
