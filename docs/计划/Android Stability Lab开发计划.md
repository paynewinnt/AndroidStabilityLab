# Android Stability Lab 开发计划

## 1. 文档目标

本文档用于定义 `Android Stability Lab` 的实施计划，统一版本顺序、阶段任务、交付边界、优先级和风险。

本文档聚焦：

- 先做什么
- 后做什么
- 每个版本如何拆阶段推进
- 当前项目应如何从执行闭环逐步推进到分析、准入、协作和集成闭环

## 2. 开发总原则

- 先做闭环，再做扩展
- 先做本地执行能力，再做本地优先分析能力
- 先做稳定数据基础，再做聚合与准入
- 不跳过 `V1 / V2` 直接建设平台化外壳
- 核心能力优先沉淀为领域对象和服务层，再由 CLI/Web 承接
- 保留性能监控能力，并将其逐步纳入任务、分析和报告链路
- 默认按本地优先、本地部署、自看自用、导出报告/JSON/产物共享建设；Web/API 是本地或团队内网共享入口，不等于完整多租户、完整 IAM 或完整在线平台已完成

## 2.1 开发状态同步规则

- 每次完成一项实际开发后，必须在同一轮同步更新对应版本计划文档中的“阶段标题”和“当前开发状态”，不能只更新代码或只更新正文摘要
- 若某项能力同时影响总计划摘要、版本子计划、测试说明或验收脚本说明，则这些文档必须一起同步，避免出现“代码状态已变、计划口径未变”
- 阶段标题中的状态短句应始终保持最新，例如“未开始 / 已完成一部分 / 已基本完成”，并同时写明“已落地内容 + 主要剩余缺口”
- 新增 CLI、Web 页面、验收入口、真实 smoke 或规则治理能力后，默认需要同步更新：
  - 总计划中的版本级摘要
  - 对应版本子计划中的阶段标题与当前开发状态
  - `tests/README.md` 中的使用或验证说明
- 如果本轮开发没有同步更新文档，应视为本轮未完整收口

## 3. 版本开发顺序与当前阶段

| 版本 | 开发目标 | 交付重点 |
| --- | --- | --- |
| V1 | 打通执行闭环 | 设备、任务、执行、异常、证据、报告 |
| V2 | 打通分析闭环 | 聚合、Top Issue、回归、归因、分析视图 |
| V3 | 打通流程闭环 | 无人值守、准入、协作、集成、Web 主入口 |

当前总阶段判断：

- `V1 / V2` 主链已基本闭环，继续作为执行、证据、分析、规则治理和准入判断的可信基础。
- `V3` 最小流程闭环已落地，当前已具备无人值守、准入中心、最小协作、本地 Web 门户、外部链路 outbox/worker/API 骨架和本地共享入口。
- 后续主线聚焦生产化治理、真实平台适配、长稳产品化、规则/证据精度、长时运行可靠性和本地共享体验收口。
- 当前共享口径以本地部署、本地 Web、报告/JSON/HTML/Markdown、分析快照、规则评审报告、准入对比和附件产物导出为主；不按完整在线审批流、完整多租户隔离、完整企业级 IAM 或完整跨组织平台验收。

### 4.1 开发摘要

`V1` 负责打通执行闭环，形成可执行稳定性任务的本地执行代理。

总文档仅保留版本级摘要：

- 阶段概览：基础重构、设备与任务底座、场景执行与异常检测、证据抓取与报告输出、稳定性收尾均已完成
- 交付重点：设备池、任务与执行实例、首批模板、基础异常检测、基础证据抓取、统一任务报告、性能监控能力接入任务链路
- 当前状态：`V1` 已形成稳定执行闭环，首批模板已从 `monkey / cold_start_loop` 扩展到 `foreground_background_loop / install_uninstall_loop / reboot_loop / standby_wake_loop`，当前主职责是继续作为 `V2 / V3` 的可信执行与证据基础，不再在总文档中混入后续版本的分析和流程能力展开
- 计划原则：先数据模型与调度，再模板与检测，最后报告与稳定性验收

## 5. V2 开发计划

版本子文档：[`Android Stability Lab V2开发计划.md`](./versions/Android%20Stability%20Lab%20V2%E5%BC%80%E5%8F%91%E8%AE%A1%E5%88%92.md)

### 5.1 开发摘要

`V2` 负责打通分析闭环，把 `V1` 的执行结果沉淀为可分析、可聚合、可回归判断的平台能力。

总文档仅保留版本级摘要：

- 阶段概览：聚合底座、分析视图、回归分析、归因与扩展证据
- 当前补充状态：`custom` 模板已支持 `uiautomator2 / adb_script / external_script` 三类业务遍历接入，结果会继续进入既有 Run / 报告 / 分析主链
- 交付重点：问题指纹与聚合、Top Issue、版本/设备/场景对比、回归结论、关键性能指标趋势分析
- 当前状态：`V2` 主链已基本闭环，已覆盖分析快照、规则治理、golden suite、规则评审报告、基线历史、准入对比，以及黑屏/freeze/system_server crash/watchdog 一等文本启发式识别、结构化 evidence、首批扩展证据引用和 artifact 来源回灌；freeze/black_screen 会优先抓 screenshot、SurfaceFlinger、input events 和 Perfetto trace，watchdog/system_server crash 会优先抓 dropbox、logcat、traces、tombstone 和 Perfetto trace，并把来源合并进 `evidence_signals / matched_sources / confirmation_level`；OOM/内存上涨/卡顿恶化启发式性能风险项和最小 evidence-aware 规则式初步归因解释已落地；规则配置入口已具备只读查看与候选编辑预览口径，可引导校验、diff、回放和评审，但规则文件仍由本地文件管理，不按完整在线规则编辑器验收；性能风险主链已支持 `PerformanceRiskThresholdConfig` 配置对象注入、scoped override 和 `config/performance_risk_thresholds.json` 本地阈值文件加载，并能输出阈值来源、命中 scope 与阈值值，剩余工作以规则精度、Web 阈值治理入口、真实阈值标定、证据结构化解析、保留策略和少量扩展证据优化为主，不能按完整智能诊断或 AI 根因诊断验收
- 计划原则：先建立可信数据与规则底座，再建设分析消费视图，最后补齐归因与高成本证据

### 6.1 开发摘要

`V3` 负责打通流程闭环，把分析平台提升为可进入团队生产流程的量产与协作平台。

- 交付重点：`7x24` 无人值守、质量门禁、本地 Web 主入口、最小团队协作、外部系统集成、报告/JSON/产物共享
- 当前状态：`V3` 最小流程闭环已落地，当前已具备无人值守 runner、准入中心、AdmissionCase、结构化准入 report payload/API/页面摘要、最小协作闭环、本地身份解析、受信 header SSO 最小接入形态、最小只读用户目录与责任检索 API、本地 outbox 事件、`CI + IM + 缺陷系统 + 提测平台` 四条正式外部链路和本地 Web 门户；质量门禁侧已具备规则治理、准入评审、基线审计、只读规则配置中心和候选编辑预览，但还不是可直接在线保存/发布规则的完整编辑器，也不是完整对外 PDF/邮件/审批流；长稳运行模板化能力已具备最小模板族画像和配置建议，可把现有稳定性模板纳入无人值守轮转、补位、隔离恢复和日报/周报链路，但仍不是完整独立长稳产品；设备池治理已具备主备设备、轮转、候补/空闲池补位、隔离和恢复探测，可支撑本地无人值守调度辅助；当前共享以本地部署、本地 Web、JSON/HTML/Markdown 报告、分析快照、规则评审报告、准入对比和附件产物为主，不按完整多租户、完整企业级 SSO/IAM、完整在线审批或跨组织报告平台验收；剩余重点已转向生产化治理、真实平台适配、长时 worker 联调、规则发布治理、团队级设备调度治理和长稳模板产品化收口
- 计划原则：先保证长期运行与准入可信，再建设协作和外部对接，逐步形成本地优先的 Web 共享入口

## 7. 依赖关系

- `V1` 不稳定，`V2` 无法做出可信聚合和回归
- `V2` 分析能力不足，`V3` 的准入和流程会变成空壳
- 问题指纹和规则治理不稳定，会直接拖垮 `V2 / V3`
- 附件治理和存储策略不明确，会直接拖垮平台长期可用性

## 8. 主要风险

- adb 稳定性不足
- 多设备并发下执行线程不稳定
- 页面层与服务层职责边界不清
- 大附件存储压力过大
- 规则质量不足导致回归和准入结论不可信

## 9. 当前建议

当前最优推进路径如下：

1. 继续把 `V1 / V2` 视为稳定基础层，优先修精度、保留策略和少量缺口
2. 当前主线按 `V3` 最小流程闭环已落地处理，优先把准入对象、身份边界、协作闭环、外部投递和本地共享入口推进到生产化可靠，而不是继续堆只读展示页
3. 所有新增核心能力优先沉淀为可复用服务和领域对象，再由 CLI/Web 承接，不把项目退化成单纯监控看板
4. 当前 Web 入口已进入“本地优先的团队共享入口边界”阶段：已支持 `team_entry` 部署模式、平台说明页、平台/ready API、`/api/manifest`、`/api/openapi.json`、`/api/users`、`/api/responsibility`、request id 和基础响应边界，以及非本地 webhook 的 `https + signing_secret` 回调安全约束；同时外部系统主链已完成 `CI + IM + 缺陷系统 + 提测平台` 四条正式接入，正式在线身份边界的最小受信 header 接入、派生用户 profile 和只读责任检索已部分落地。本地部署和本地 session / `X-ASL-Actor` 路径仍可用，查看者默认看到同一份数据，身份主要约束写操作、审计和责任检索；当前设备池治理也应按本地无人值守调度辅助理解，已有主备设备、轮转、补位、隔离与恢复探测，但不等于完整团队设备预约、配额、审批或排班平台；完整企业级 SSO/IAM、真正跨系统责任自动同步、团队级设备调度治理、真实平台适配和长时 worker 联调仍应继续后置建设
5. 长稳运行模板化能力当前按“已具备最小模板族画像和配置建议”验收：它可以指导把 `monkey / cold_start_loop / foreground_background_loop / install_uninstall_loop / reboot_loop / standby_wake_loop / custom` 纳入无人值守长稳任务，但仍需补独立长稳模板入口、参数推荐沉淀、专项验证样本和更正式配置面，不能写成完整长稳产品已完成
6. 版本测试报告 / 准入报告当前按“结构化 report payload + Web/API 摘要 + 本地报告、审计和对比产物跳转”验收；本地部署、报告/JSON/HTML/Markdown 导出仍是共享主口径，不能写成已完成对外 PDF、邮件分发或正式审批流
