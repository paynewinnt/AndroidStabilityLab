# Android Stability Lab V1开发计划

## 1. 文档目标

本文档基于主计划文档第 4 章 `V1` 开发计划整理，目标是在不突破主文档版本边界的前提下，将 `V1` 计划扩展为可直接继续拆分排期、研发实现、测试验收和项目跟踪的版本子文档。

`V1` 的核心任务是打通执行闭环，形成可执行稳定性任务的本地执行代理。

## 1.1 状态维护规则

- `V1` 任一阶段有实际开发落地后，必须在同一轮同步更新本文件对应阶段的标题状态短句和“当前开发状态”
- 更新时不能只改阶段正文，不改标题；也不能只改总计划摘要，不改本子计划
- 阶段标题应明确写出“当前状态 + 已落地内容 + 主要剩余缺口”，保证只看标题也能判断推进程度

## 2. 版本范围与目标

### 2.1 版本定位

`V1` 聚焦本地执行能力建设，完成“设备接入 -> 任务创建 -> 场景执行 -> 异常识别 -> 证据抓取 -> 报告输出”的闭环。

### 2.2 版本目标

- 建立可维护的设备池与设备状态管理能力
- 建立基础任务系统和执行实例模型
- 接入首批稳定性执行模板
- 建立基础异常检测与 issue 记录能力
- 在异常发生时自动采集基础证据
- 在任务结束后输出统一报告
- 保留现有性能监控能力，并将其接入任务执行链路

### 2.3 版本边界

本版本只解决执行闭环问题，不包含以下内容：

- 不建设中心化分析平台
- 不建设问题聚合、Top Issue、回归判断等 `V2` 能力
- 不建设无人值守准入、协作流转、Web 主入口等 `V3` 能力
- 新增核心能力应优先沉淀为可复用模块

## 3. 总体推进策略

### 3.1 推进原则

- 先落数据模型与执行底座，再扩展执行模板
- 先保证单机多设备任务可跑通，再做稳定性补强
- 先保证异常与证据链闭环，再优化展示和体验
- 现有性能采样、图表和导出能力优先复用，不做大范围重写

### 3.2 建议推进顺序

1. 建立设备、任务、执行实例等基础数据模型
2. 建立任务调度与执行生命周期管理
3. 接入首批模板并打通执行参数
4. 接入异常检测与 issue 记录
5. 打通证据抓取、归档与报告输出
6. 最后补齐重试、超时、掉线恢复等稳定性能力

## 8. 依赖与协同要求

### 8.1 关键依赖

- 现有性能采样能力可被稳定复用
- adb 与设备连接链路具备可接受稳定性
- 现有项目代码结构允许逐步抽离核心能力

### 8.2 前后置关系

- 阶段 1 完成前，不建议并行推进大量模板开发
- 阶段 2 完成前，异常检测与证据抓取无法形成稳定挂载点
- 阶段 3 完成前，报告输出只能做静态结构，无法验证完整内容
- 阶段 4 完成前，`V1` 不具备可验收的闭环价值

### 8.3 角色协同建议

- 研发优先完成模型与执行主链路
- 测试从阶段 2 起同步建设链路级用例
- 项目管理按里程碑跟踪依赖、风险和版本边界

## 9. 主要风险与应对

| 风险 | 影响 | 应对建议 |
| --- | --- | --- |
| adb 稳定性不足 | 任务中断、设备状态不可信 | 优先建立掉线识别、重连和失败标记机制 |
| 多设备并发执行不稳定 | 执行结果不可靠，阻塞后续验收 | 尽早建立并发压测与资源隔离策略 |
| 页面层与服务层耦合过深 | 重构成本高，迭代速度慢 | 新逻辑统一沉淀为服务层和领域对象，CLI/Web 只作为入口 |
| 证据附件体积过大 | 存储与归档压力增大 | 提前约定附件目录、保留策略和压缩策略 |
| 异常规则质量不足 | issue 记录噪声大，影响后续分析 | 首批规则只覆盖高置信问题，逐步扩展 |

## 10. 验收标准

### 10.1 版本验收核心标准

- 可管理多个设备并识别基础状态
- 可创建任务并在目标设备上生成执行实例
- `Monkey` 与 `冷启动循环` 模板可成功执行
- 基础异常可被识别并形成统一记录
- 异常发生后可自动抓取基础证据
- 任务结束后可生成统一报告
- 现有性能监控能力已进入任务执行与报告链路

### 10.2 建议测试覆盖

- 单设备正常链路执行
- 多设备并发执行
- 设备中途掉线
- 执行超时
- 异常触发后的证据抓取
- 报告生成完整性
